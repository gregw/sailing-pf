HPF Documentation

The Problem with Elapsed-Time Analysis

Rating yacht performance from elapsed times sounds straightforward, but the data is deeply ambiguous. For any single race result, we do not know:

  • Race length — the distance sailed varies with course set, tidal current, and the actual wind shifts encountered. Two races on the "same course" can differ significantly in effective length.
  • Weather conditions — wind strength and direction, sea state, and tidal set affect different designs very differently. A light-air race favours some boats; a rough beat with strong tide favours others. The same boat can look fast or slow depending purely on conditions.
  • Course type — a windward-leeward course tests upwind and downwind performance; a harbour race tests tacking angles in chop; a coastal race is dominated by navigation and current management. No single factor applies equally to all.
  • Extraordinary events — a boat that breaks a halyard, fouls a mooring, or is becalmed in a localised hole looks poor against its history. Weather and tide gates can split a fleet so that half the boats experience entirely different conditions from the other half.
  • Crew and configuration — a boat racing short-handed, with a pickup crew, with the sails from the wrong season, or with an owner learning the boat will look different from the same hull sailed by a skilled regular crew. We cannot distinguish boat speed from crew speed from elapsed times alone.
  • Luck — starting position, tactical choices, and plain fortune introduce noise that no amount of statistical sophistication can fully remove.
  • Poor data quality — competitors enter much of the data, so that there are many typos, abbreviations, sponsors names, punctuation differences, duplicate entries, etc. This makes it difficult to identify what is actually the same boat, or just a similar boat with a similar name (only so many names starting with J or X). There are also some extraordinarily wrong times entered.

Even if every one of those factors were stable, sample sizes are tiny. Most boats in a club fleet do perhaps 20–50 races per year, often split across multiple series with changing competition. A typical boat in this database may have 50–200 recorded finishes spanning several years. In the context of statistical inference, where each observation is contaminated by all the unknowns above, this is a very thin basis for a reliable estimate of anything. The HPF for a boat with fewer than about 20 finishes should be treated as highly tentative.

The optimiser mitigates these problems — outlier down-weighting suppresses individual bad results; the asymmetry principle discounts apparently extraordinary fast finishes; the weighted median reference time reduces the influence of large-fleet outliers — but it cannot eliminate the underlying uncertainty. The confidence weights and per-race residual charts are the honest acknowledgement of this: a low-weight HPF and a wide residual scatter mean we genuinely do not know what this boat should be handicapped at from its history alone.

Measurement certificates help, but not primarily by providing accurate individual factors. Their main value is as a common reference: IRC, ORC, and AMS are all widely used, and a boat's certificate value in any of those systems gives us a plausible starting point that is roughly comparable across the fleet, across clubs, and across years. By anchoring the HPF optimisation to certificate-derived Reference Factors, we ensure that the resulting factors are at least expressed in a scale similar to other systems — so an HPF of 0.950 means roughly what IRC 0.950 means, which is something other clubs and handicappers can interpret. Without that anchor, HPF values would be identified only up to an arbitrary global scale factor and would be meaningless outside the specific fleet used to compute them.

Pages

Tasks Page

Controls data import and processing. Tasks are listed in the order they run during a scheduled run, which matters: analysis requires races to be imported first; reference-factors requires analysis; hpf-optimise requires reference factors. Any task can also be run individually on demand. Scheduling is configured by day-of-week and time-of-day, and runs all "in schedule" tasks sequentially.

Data Browser

Browse and inspect all entities in the database across four tabs: Boats, Designs, Clubs, and Races. Each tab has search, sort, and pagination. Click any row to expand the full JSON record. For boats, the expanded view also shows Reference Factors and HPF detail. For races, the expanded view shows a division chart.

The checkbox column on Boats and Designs allows selecting multiple records for a merge operation — useful for consolidating duplicate entries created by name or sail-number variations across source systems.

Handicap Analysis

Visualises the empirical conversion network between handicap systems (IRC, ORC, AMS) across years and variants. Each edge represents a linear regression computed from boats holding certificates in both systems. Click an edge to see the scatter plot and a conversion lookup table. The HPF Run Quality panel appears here after the optimiser has run.

Boat Comparison

Select one or more boats and plot their back-calculated per-race factors over time on a single chart. Useful for comparing how two or more boats have performed relative to each other historically, or for checking whether a boat's factor is trending.

A handicap calculator is provided that uses the RF and HPF for all the compared boats to scale a handicap allocated to one boat to the others.

Design Comparison

Select two design classes and compare their corrected elapsed times across shared race divisions. Each shared division provides a direct A-vs-B comparison under identical conditions. A best-fit line close to the RF-ratio reference line indicates good agreement between the theoretical conversion (from certificates) and the empirical result.

A handicap calculator provides both RF-ratio and best-fit-based handicap conversion from one design to the other.

Data Browser

Boats Table

Lists all boats in the database. Boats are identified by sail number and name; because neither is guaranteed unique, two source records for the same physical boat may appear as separate rows. Use the merge function (checkboxes → merge) to consolidate them.

The RF and HPF columns are colour-coded: green indicates high confidence (weight close to 1.0), red indicates low confidence. A dash (—) means no value has been computed for this variant.

Boats › ID

A unique stable identifier constructed from the normalised sail number, boat name, and design ID. The format is sailnum-name-designid (e.g. AUS1234-raging-tp52). IDs are never derived from SailSys or TopYacht internal IDs, which are stored only in alias tables for ingestion purposes.

Boats › Sail Number

The sail number as recorded in the primary source. Country prefixes (e.g. AUS, GBR) are preserved. Sail numbers are not unique — different boats may share the same number, and the same boat may have its number recorded differently across sources.

Boats › Name

Boat name from the primary source. Aliases and name variations are reconciled during import using configurable fuzzy-matching; the canonical name shown here is whichever was first recorded or promoted by an admin merge.

Boats › Design

The design (class) ID linking this boat to a Design record. Derived from the ORC Class field where available, or from the SailSys make/model fields otherwise. Click to see the associated design record in the Designs tab.

Boats › Reference Factor (RF)

The IRC-equivalent handicap derived from this boat's measurement certificates (IRC, ORC, AMS) via the conversion network. Three variants are computed independently: spinnaker (spin), non-spinnaker (NS), and two-handed. The value shown in the table is the spin variant; the full breakdown is in the boat detail panel.

Weight reflects confidence: 1.0 = derived directly from a certificate in the target year; lower = inferred via multi-hop conversion, design-level aggregation, or race co-participation propagation. See the colour scale: green = weight close to 1.0, red = low weight.

Boats › HPF

The Historical Performance Factor — the back-calculated handicap from the ALS optimisation averaged over all recorded race entries for this boat. The weight reflects the quantity and quality of contributing races; a boat with few races or highly variable results will have low HPF weight. Colour scale matches RF.

Boats › Overall (Performance Profile Score)

The normalised area of the performance profile pentagon — a single number in [0, 1] summarising how the boat compares to the active fleet across five dimensions over the last 12 months. A score of 1.0 would mean the boat ranks first in the fleet on every spoke simultaneously; 0.0 means last on every spoke. In practice most boats sit between 0.1 and 0.5. Click the boat row to see the full pentagon chart and per-spoke details. See Performance Profile for spoke definitions.

Only boats with at least 3 distinct races in the last 12 months receive a profile score. Boats with fewer races are excluded from the ranking pool entirely, so the score reflects only the regularly active fleet.

Boats › Club

The home club identifier (website domain). A boat may be associated with multiple clubs across different source systems; the value shown is the primary club from the most authoritative source.

Boats › Finishes

Total race finishes recorded in the database across all clubs and series. Click the number to switch to the Races tab filtered to show only races for this boat.

Boats › Excluded

Tick to exclude this boat from all analysis, charts, and HPF calculations. Use for test entries, ghost boats, or data errors. Excluded boats are still visible in the data browser (greyed out) unless "Show excluded" is unticked.

Designs Table

Groups boats by design class. Design-level factors are used as fallback anchors for boats in the same class that lack individual certificates.

Designs › ID

Designs › Canonical Name

The design name used in all reports and charts — typically the ORC class name or the most common name found across source systems.

Designs › Reference Factor (RF)

Design-level IRC-equivalent RF aggregated across all certificated boats of this class. See Boats › RF for weight interpretation. A design RF is only as reliable as the certificate data available for boats of this class.

Designs › Makers

Designs › Boats

Races Table

Individual race results imported from SailSys, TopYacht, and BWPS. A single race may have multiple divisions (different handicap systems or fleet splits for the same start). Click a race row to see the full JSON record and the division chart.

Races › ID

Unique race identifier: clubDomain–date–nnnn (e.g. myc.com.au-2024-11-06-0001). The four-digit suffix distinguishes multiple races at the same club on the same date. IDs are never reused and are independent of source system internal IDs.

Races › Club

The club that ran this race. Click to filter the Races tab by this club.

Races › Date

Race date in ISO format (YYYY-MM-DD).

Races › Series

The series this race belongs to (e.g. Wednesday Night Racing, Autumn Series). Click to filter the Races tab to all races in this series — useful for browsing a season's results or navigating prev/next within a series.

Races › Race Name

The race name or number within its series as recorded in the source system (e.g. "Race 3", "November 6").

Races › Finishers

Total finishers across all divisions in this race.

Races › Ref Time

T₀ — the estimated elapsed time a 1.000 HPF boat would have taken to complete the course, computed as the weighted median of (elapsed ÷ HPF) across all boats with reference factors in this race. Only available after the HPF optimiser has run. For multi-division races, each division's T₀ is shown separately (e.g. Div A: 1:19:59 / Div B: 1:07:07).

The weighted median is used rather than the mean because elapsed-time distributions are right-skewed — slow outliers from gear failure, bad wind holes, or tactical errors are common and should not distort the baseline.

Races › Excluded

Excluded races are not used in HPF calculations. Individual races can be excluded where results are known to be unreliable (e.g. abandoned race, severe weather split affecting only part of the fleet).

Race Division Chart

Shows the elapsed time and corrected times for each finisher in the selected division, plotted against their HPF on the x-axis. Three lines are shown:

  • Elapsed — raw elapsed time. Should slope upward from left to right: faster boats (lower HPF) finish in less time.
  • HPF corrected — elapsed divided by HPF. If the HPF is correct for this race, this line should be approximately flat (all boats equal-time).
  • RF corrected — elapsed divided by the Reference Factor. Shows how the certificate-based correction compares to the HPF correction.

A flat corrected-time line confirms the factor was appropriate for this race. A sloping line indicates the faster or slower boats systematically out- or under-performed their factor — possibly due to conditions favouring one end of the fleet, or an outlier performance.

Error bars on the corrected lines (enable with the "Error bars" checkbox) show ±95% confidence intervals derived from the RF weight. See Error Bars for the math.

Boat Detail

Click any boat row to expand the detail panel. Below the JSON record, two tables are shown: Reference Factors and HPF Detail.

Reference Factors Table

Shows the IRC-equivalent Reference Factor for each handicap variant (Spin, Non-Spin, Two-Handed), alongside the confidence weight and the generation (number of propagation hops from a direct certificate). These are the anchor values fed into the HPF optimiser.

Reference Factors › Variant

Spin (with spinnaker), Non-Spin, or Two-Handed. Each variant is computed independently because ORC, AMS, and IRC all issue separate certificates for different sailing configurations.

Reference Factors › Value

IRC-equivalent TCF (Time Correction Factor) for this variant, converted to the target IRC year via the empirical conversion network. A value of 1.000 represents the hypothetical reference boat.

Reference Factors › Weight

Confidence weight for this factor, between 0 and 1. The weight starts at 1.0 for a direct certificate in the target year and degrades through each source of uncertainty:

Source Weight
Direct certificate, target year 1.00 (base)
ORC club certificate × 0.90
Windward/leeward cert (ORC WL) × 0.80
Design-level aggregate × 0.85
Race co-participation propagation × 0.70
Each year from target (certificate age) × 0.80 per year
Each ConversionGraph hop × edge weight (R²-derived)

The weight is treated as inversely proportional to variance in log space. See Error Bars for the full derivation.

Reference Factors › Gen

Generation: how many propagation steps removed from a direct certificate. Generation 0 = direct certificate (or one-hop graph conversion from a direct certificate). Generation 1 = design-level aggregate of gen-0 boats. Generation 2+ = race co-participation propagation from gen-1 boats, and so on. Higher generations carry more uncertainty and lower weights.

HPF Detail Table

Shows the optimised Historical Performance Factor for each variant alongside the corresponding Reference Factor, for easy comparison. The columns mirror the HPF summary on the Boat Comparison page.

HPF Detail › Variant

Spin, Non-Spin, or Two-Handed — matching the RF variant. HPF is computed separately for each variant using only races run under that handicap configuration.

HPF Detail › RF

The Reference Factor for this variant — the certificate-derived anchor value. See Reference Factors › Value.

HPF Detail › HPF

The Historical Performance Factor — the back-calculated handicap from the ALS optimisation. A value close to RF means the boat's race history is consistent with its measurement certificate. A persistent gap may indicate the certificate is stale, the boat has been modified, or the fleet it races against has systematically drifted.

HPF Detail › HPF Weight

Confidence weight for the HPF estimate, derived from the number and quality of contributing race entries. More races and more consistent results yield higher weight.

HPF Detail › Delta

HPF minus RF in the same units. Near zero = race history is consistent with the certificate. Positive delta = the boat races slower than the certificate predicts (HPF > RF). Negative delta = the boat races faster (HPF < RF). Large deltas on high-weight boats are worth investigating.

HPF Detail › Races

Number of race division entries contributing to this HPF estimate. More races generally means higher confidence, but consistency across races matters more than sheer count.

Performance Profile

A pentagon radar chart showing how this boat compares to the active fleet across five fleet-relative percentile scores, each computed from the last 12 months of race data. Each spoke ranges from 0 (worst in fleet) to 1 (best in fleet), where rank is relative to all boats with at least 3 distinct recent races. The overall score shown below the chart is the normalised polygon area of the pentagon (see Boats › Overall).

The five spokes, in pentagon order:

Spoke What it measures How it is scored
Frequency How regularly the boat races Percentile rank of distinct race count in the last 12 months. Higher race count = better rank. A boat racing once a fortnight will outrank one racing only a handful of times.
Consistency How tightly the residuals cluster around zero Percentile rank of the weighted sum of squared residuals (lower raw value = better rank). A boat with small, stable residuals scores well; wide scatter scores poorly. Scores "how predictable is this boat race to race".
Diversity How broadly the boat races across the fleet Percentile rank of distinct opponents raced against in the last 12 months. A boat that competes in large, varied fleets scores higher than one racing only against the same small group each week.
NonChaotic Whether performance variation is weather-driven rather than boat/crew-driven Percentile rank of the Pearson correlation between |residual| and the race's fleet dispersion (weighted IQR of residuals across the whole fleet that day). High correlation means the boat goes wide when the whole fleet goes wide — its inconsistency is due to variable conditions, not intrinsic unreliability. Boats with insufficient dispersion data (fewer than 5 paired observations) receive a score of 0.
Stability Whether the boat's trend is level or declining Percentile rank of an asymmetric slope penalty computed from a weighted linear regression of residuals against race date. A level trend (slope ≈ 0) incurs no penalty and ranks best. An improving trend (negative slope — boat getting faster relative to the fleet) incurs half the penalty of an equivalent declining trend. A declining trend (positive slope — boat getting slower) is penalised most heavily, since it suggests the current HPF may already be out of date.

The polygon area formula normalises the area so that a perfect pentagon (all spokes = 1) gives an overall score of 1.0 and a degenerate case (any spoke = 0) collapses the adjacent triangles, reducing the score.

Per-Race Residuals Chart

Scatter plot of per-race residuals in log space over time. The residual for one race entry is:

residual = log(elapsed × HPF) − log(T₀)

A positive value means the boat finished slower than the fitted model predicted; negative means faster. Points near zero indicate races where the boat performed close to its HPF. Colour intensity reflects the entry weight used in the optimiser — faint points are outliers that were down-weighted and contribute little to the HPF estimate. A systematic trend (residuals growing more positive over time) may indicate the boat is slowing relative to the fleet.

Tasks

Task List

The task list shows all available import and processing tasks. Tasks are run in table order during a scheduled run, which is significant: later tasks depend on the output of earlier ones. The full dependency chain is:

sailsys-races / orc / ams / topyacht / bwps → analysis → reference-factors → build-indexes → hpf-optimise

Any task can also be triggered individually. The status column updates in real time during a run. A "Stop schedule" button appears while a schedule is in progress.

Tasks › Task

The task name and run mode: api means the task fetches from an external network source (may be slow or fail if the source is unavailable); run means it computes from data already in the local store (always fast).

Tasks › Status

idle — the task is not running and is ready to start. running — currently executing; logs appear in the server console.

Tasks › Run

Click to run the task immediately, regardless of schedule. For sailsys-races, an optional starting race ID field allows resuming from a specific ID rather than scanning forward from the last processed ID — useful after a gap in collection.

Tasks › In Schedule

Tick to include this task in the automatic scheduled run. Tasks not ticked are skipped during scheduled runs but can still be triggered manually.

Tasks › On Start

Tick to run this task automatically each time the server starts. Useful for tasks like build-indexes which are fast and keep in-memory state fresh after a restart.

Task: Import SailSys Races

Fetches race results from the SailSys API, scanning forward from the configured start ID. Imports race metadata, divisions, finishers, elapsed times, and certificates inferred from the handicap used for scoring (handicapCreatedFrom). Supports a local file cache for bulk re-processing without network calls.

Unknown boats are resolved by fuzzy name matching (Jaro-Winkler similarity) or created as new records. The SailSys boat API is called on demand for design information.

Task: Import ORC Certificates

Downloads the ORC Australian certificate feed from data.orc.org. For each certificate, fetches the detail page to extract GPH and converts to TCF via 600/GPH. Detects non-spinnaker (NS), double-handed (DH), and club variants from the CertType and FamilyName fields. Creates or updates Boat, Design, and Certificate records. Idempotent: re-running does not duplicate records.

Task: Import AMS Certificates

Scrapes AMS certificate listings from raceyachts.org. Creates Certificate records with system="AMS". Includes non-spinnaker and two-handed variants.

Task: Import TopYacht Races

Scrapes race results from TopYacht club result pages listed in clubs.yaml. Parses group captions to detect handicap system and variant flags — for example, a caption of "Div 1 ORC NS WL results" sets system=ORC, nonSpinnaker=true, windwardLeeward=true. Multi-page results (same race, multiple handicap systems) are merged into a single Race record with multiple divisions. PHS handicap values are excluded; only elapsed times are kept from PHS divisions.

Task: Import BWPS Races

Imports BlueSail Worldwide Performance System race results from the Cruising Yacht Club of Australia. IRC-based races providing elapsed times and the AHC used for scoring.

Task: Analyse Certificates

Builds the ConversionGraph — a network of empirical linear fits between handicap systems, years, and variants. For each boat holding certificates in two different (system, year, variant) combinations, the pair of TCF values contributes one observation. Observations are grouped by pair type and a weighted linear regression is fitted. Edges with R² < 0.75 are discarded. The resulting graph enables conversion between any two connected nodes via shortest weighted path.

Task: Calculate Reference Factors

Traverses the ConversionGraph (DFS) from each boat's certificates toward the target IRC node, computing an IRC-equivalent Reference Factor. Three variants (spin, nonSpin, twoHanded) are computed independently. After certificate-based factors are assigned, the task iterates: design-level aggregation and race co-participation propagation extend coverage to boats without direct certificates, up to 20 iterations or until no new factors are produced.

Task: Build Indexes

Rebuilds the in-memory navigation indexes: boatId → race IDs, designId → boat IDs, seriesId → boat IDs. These indexes are volatile (lost on server restart) and must be rebuilt after any import that adds new boats or races. The task is fast (<1 second) and is typically ticked "on start".

Task: HPF Optimise

Runs the Alternating Least Squares (ALS) optimisation to produce Historical Performance Factors for all boats in the database. The optimisation solves jointly for:

  • HPF per boat — one stable value across all races in scope
  • T₀ per race — the estimated reference time for each race

A regularisation term (controlled by λ) pulls each boat's HPF toward its Reference Factor, weighted by RF confidence. The outer loop recomputes entry weights from residuals (outlier down-weighting) and re-runs the inner ALS until weights stabilise. See Processing Pipeline steps 13–19 for the full mathematical description.

Schedule & Configuration

Schedule Days

Days of the week on which the scheduled run executes. Multiple days can be selected. No days selected = schedule disabled. Saturday/Sunday night runs (e.g. 03:00 Sunday) are typical for low-traffic windows after weekend racing.

Schedule Time

Time of day (server local time) at which the scheduled run begins. All "in schedule" tasks run sequentially from this start time. A full run including network imports and optimisation typically takes 5–30 minutes depending on dataset size and network speed.

Target IRC Year

The IRC certificate year used as the target node in the ConversionGraph — all Reference Factors are expressed as IRC-equivalent TCF for this year. Leave blank to auto-detect from the most recently issued IRC certificates in the database. Set explicitly when fresh IRC data has not yet been imported but you want to lock to the current year, or when testing with historical data.

HPF Parameters

These parameters control the ALS optimisation algorithm. Changes take effect on the next run of the hpf-optimise task. The HPF Run Quality panel on the Analysis page shows whether the current settings achieve convergence and what fit quality results.

Lambda (Regularisation)

The regularisation strength λ controls how strongly each boat's HPF is pulled toward its Reference Factor. The per-boat pull is weighted by that boat's RF confidence weight, so well-certificated boats are anchored more tightly than those relying on propagated factors.

  • λ = 0 — pure race-data optimisation; HPF values float freely. Only useful if all boats have strong RF anchors, otherwise the solution is unidentified up to a global scale factor.
  • λ = 10 (typical default) — moderate anchor; HPF can deviate from RF but is pulled back.
  • Higher λ — stronger trust in certificates over race results. Increase if HPF values drift far from certificates for well-certificated boats.

Convergence Threshold

Inner loop convergence criterion. The ALS inner loop stops when the maximum change in any log(HPF) across a single iteration falls below this value. Default ~0.0001. Smaller = tighter solution but more iterations per outer cycle. Increase if inner convergence is very slow; decrease for maximum precision.

Max Inner Iterations

Maximum number of ALS iterations (alternating T₀ solve then HPF solve) per outer reweighting cycle. Increase if the quality report shows "inner loop did not converge". Typical convergence occurs within 20–50 iterations for well-conditioned problems.

Max Outer Iterations

Maximum number of outlier reweighting cycles. Each outer cycle: recomputes entry weights from current residuals, then re-runs the inner ALS to convergence. Typically 3–5 outer iterations suffice. Increase if the quality report shows "outer loop did not converge".

Outlier K (IQR Multiplier)

Controls the aggressiveness of outlier down-weighting via a Cauchy weight function. An entry with residual magnitude greater than k × IQR is treated as an outlier and assigned reduced weight (but never excluded entirely). Default ~2.0. Lower = more aggressive suppression (useful for noisy datasets); higher = more permissive (useful when the fleet is homogeneous and outliers are genuine).

Asymmetry Factor

Extra penalty multiplier applied to entries where the boat finished faster than the model predicts (negative residuals). Reflects the asymmetry principle: it is essentially impossible for a boat to genuinely sail faster than its potential in a small fleet; an apparently outstanding result is more likely to reflect poor performance by the rest of the fleet or a favourable weather/tide gate.

Default ~1.5 means unexpectedly fast entries are down-weighted 1.5× harder than equivalently slow entries. Higher values are more conservative about crediting apparently outstanding performances.

Outer Damping Factor

Blend fraction for weight updates between outer reweighting cycles (0–1). The updated weight is a blend of the previous weight and the weight recomputed from current residuals: new_weight = damping × computed + (1 − damping) × old. Lower values damp oscillation at the cost of slower convergence. Default 0.5. Set to 1.0 for no damping (may oscillate on poorly-conditioned problems).

Outer Convergence Threshold

Outer loop convergence criterion — maximum allowed change in any entry weight between consecutive outer cycles. Default 0.01. Increase (e.g. to 0.3) if the outer loop consistently fails to converge within the allowed iterations; this trades precision for tractability. Decrease for tighter convergence at the cost of more outer cycles.

Handicap Analysis

Handicap Conversion Network

A graph visualising the empirical linear relationships between handicap systems (IRC, ORC, AMS) across years and variants. Each node represents one (system, year, variant) combination — for example "IRC spin 2024" or "ORC NS 2023". Each edge is a linear fit computed from boats that hold certificates in both endpoint systems.

Edge colour intensity reflects R² — darker edges have stronger fits with more predictive power. Hover over an edge to see the R², sample count, and conversion slope. Click an edge to load the scatter plot and conversion table for that pair.

The network is built by the analysis task. A minimum R² of 0.75 is required for an edge to be included. Year-to-year edges within the same system capture IRC certificate reissue inflation across seasons.

Analysis Scatter Plot

Scatter plot of paired handicap observations for the conversion selected in the dropdown (e.g. "IRC spin 2024 → ORC spin 2024"). Each point is one boat with certificates in both systems. The regression line shows the best-fit linear conversion; points beyond 2.5σ are excluded from the fit (but still plotted). The optional "Error band" checkbox shows a ±1σ envelope around the regression line.

A tight cluster near the regression line indicates a reliable conversion — these two systems track each other closely. A wide scatter indicates poor inter-system agreement and the resulting edge will have lower weight in the ConversionGraph.

Conversion Table

A lookup table applying the fitted linear conversion over a configurable range of input values. Set the Min, Max, and Step values, then click Update to regenerate. Useful for manually converting a known handicap from one system to another, or for sanity-checking that the fitted conversion is sensible at the extremes.

Conversion Table › x

Input value in the source handicap system (e.g. IRC TCF). Varies from Min to Max in steps of Step.

Conversion Table › Predicted y

Predicted output value in the target handicap system via the fitted linear conversion y = a·x + b.

Conversion Table › Weight

A proportional bar showing the effective density of training observations near this x value. Values near the centre of the training range have high weight; values at the extremes (extrapolations) have lower weight and should be treated with more caution.

HPF Run Quality

Summary statistics from the most recent HPF optimisation run, displayed on the Analysis page. These help assess whether the current configuration is appropriate and whether the results can be trusted. All statistics are from the last completed run of hpf-optimise.

Convergence

Whether both the inner ALS loop and the outer reweighting loop converged within their allowed iteration limits. Non-convergence means the solution may be suboptimal:

  • Inner did not converge — increase Max Inner Iterations or relax Convergence Threshold.
  • Outer did not converge — increase Max Outer Iterations, raise Outer Convergence Threshold, or reduce Outer Damping Factor to damp oscillations.

Scale

The number of boats, race divisions, and individual race entries used in the optimisation. Larger numbers generally mean more reliable results. If the counts are much lower than expected, check that build-indexes has run after the most recent import.

Fit Quality

Median absolute residual and spread (IQR of residuals) across all race entries. Residuals are in log space; a median absolute residual of 0.02 means the typical entry's elapsed time differs from the model by about 2%. Lower values indicate a better fit. High spread suggests either highly variable race conditions or a dataset with many outlier races.

Outliers

The number of entries assigned reduced weight due to large residuals, and the number of race divisions with high internal spread (IQR/T₀ above threshold). These are down-weighted rather than excluded, so they still contribute to the optimisation but with less influence. A high outlier count may indicate noisy source data or an Outlier K setting that is too permissive.

Confidence

Median boat confidence score across all boats in the optimisation, reflecting the quantity and quality of race data per boat. A score near 1.0 means most boats have well-supported HPF estimates; a low median score means many boats have few races or highly variable results. Individual boat confidence scores are shown in the HPF Detail table.

Boat Comparison

Back-Calculated Factor Comparison Chart

Scatter plot showing the back-calculated factor for each selected boat over time. Each point is one race result — the handicap that boat would have needed to be equal-corrected-time with a 1.000 reference boat in that specific race. Hover over a point for race details (date, club, division, elapsed time).

Two optional trend overlays help identify patterns:

  • Linear trend — a weighted regression line per boat, showing systematic drift over the full history.
  • Sliding average — a trailing N-finish weighted average, showing recent form relative to longer history.

The Variant selector controls which handicap configuration's data is shown. The "All available" checkbox toggles whether the boat selector shows all boats or only boats that have raced with every boat currently in the comparison.

Handicap Calculator

Enter a handicap for one boat or design; the others are automatically scaled by HPF (or RF) ratio. On the Boat Comparison page, the suggested handicap for boat B given a known handicap for boat A is:

handicap_B = handicap_A × HPF_B / HPF_A

On the Design Comparison page, two suggestions are shown:

  • By RF ratio — scales by the ratio of design-level Reference Factors. More reliable when both designs have strong certificate coverage.
  • By best fit — applies the direct linear fit from the scatter plot. More accurate when the designs race very differently from IRC expectations, or when the RF ratio is poorly calibrated.

Design Comparison Chart

Scatter plot comparing the corrected elapsed times of two design classes across shared race divisions — divisions where at least one boat from each class competed on the same day and course. Blue = Design A, amber = Design B.

The x-axis is the boat's HPF (or RF if HPF is not yet available); the y-axis is corrected elapsed time in minutes. Two reference lines are shown:

  • Best-fit line — linear regression through the scatter.
  • RF projection — the expected relationship if both designs' corrected times matched their reference factors exactly.

A good comparison for handicap cross-referencing shows two tight clusters that overlap in the corrected-time dimension, with a best-fit line close to the RF projection. Divergence may indicate one design systematically out- or under-performs its certificate in the conditions typical of the shared races.

Technology

  • Backend: Java 21, Jetty 12.1 embedded (plain Jakarta servlets — no Spring)
  • Data store: Jackson JSON/YAML files (file-per-entity persistence)
  • HTTP client: Jetty HttpClient (API calls, web scraping)
  • HTML parsing: JSoup (TopYacht scraping)
  • Fuzzy matching: Apache Commons Text (Jaro-Winkler similarity)
  • Frontend: Plain JavaScript, Plotly.js for interactive charts
  • Deployment: Cloud Run or Fly.io (scale-to-zero)

The implementation and documentation for this project were largely created using AI, so significant human review is required.

Status

Phase 1 complete. Data ingestion (six importers: SailSys boats/races, TopYacht, ORC, AMS, BWPS), reference network construction (empirical conversion graph, DFS-based reference factor computation), and an admin webapp (data browser, analysis charts, import scheduling) are all operational.

Phase 2 (HPF optimisation) is not yet implemented. The alternating least squares HPF optimisation described in the Processing Pipeline steps 13–19 remains to be built.
